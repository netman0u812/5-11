Context
Why This Program Exists
THE PROBLEM
Ungoverned Partner Mesh
Conventional third-party connectivity is bilateral and accumulative — each partner produces a VPN tunnel, a firewall rule, and an informal agreement. Over time this becomes an ungoverned mesh with no structural isolation guarantees.
CVS SCALE
15+ Acquisition Domains
Caremark, Omnicare, Aetna, Oak Street Health, and a dozen more integration domains. Partner connectivity requests arrive across organizational boundaries with inconsistent tooling, naming conventions, and inspection standards.
THE FIX
Per-Tenant Mediation Model
3PLZ interposes a mandatory mediation layer. Every partner connects to a routing-isolated tenant. Every session transits a managed proxy and inline inspection plane. The platform is the sole path, the sole enforcement point, and the complete audit record.

The partner obtains no direct routing path to any CVS network. CVS exposes no internal addressing to any partner. The 3PLZ platform is what you are being evaluated as.

Architecture
The 3PLZ Mediation Model
Connectivity model:   Partner   →   3PLZ Tenant   ↔   Enterprise    ·    No direct partner↔enterprise routing at any layer.
01 TENANT ISOLATION
Structural, not policy-based
Each partner occupies a dedicated Virtual Router domain. No cross-tenant route exists. A misconfigured security policy rule cannot create cross-tenant reachability.
02 CONNECTIVITY
IPSec IKEv2 VTI tunnels
Partners connect via IKEv2 VTI. IKE anchor in Internet (untrust) zone. Decapsulated partner traffic emerges in a per-partner zone on the platform firewall.
03 TRANSLATION
Dual-Proxy or SNAT-to-SNAT
Bidirectional L3 translation abstracts partner and enterprise address spaces. Neither endpoint sees the other's real addressing. Key custody stays at the platform.
04 INSPECTION PLANE
Per-tenant inline DLP / Threat Prevention
Decrypted in-flight traffic — TLS and SSH — is steered to a CVS-designated per-tenant inspection container. Platform delivers plaintext payload only; no session keys leave the vendor edge.

CR-01
23% — HIGHEST STRUCTURAL RISK
Per-Tenant mTLS CA Isolation
WHAT WE REQUIRE
Each tenant receives an independent intermediate certificate authority. A partner certificate issued by Tenant A's CA must be rejected at the Tenant B proxy listener at the trust-validation layer — not at policy, not at routing.
THE DISTINCTION
Structural rejection means the platform cryptographically cannot accept the certificate regardless of any policy path. A misconfigured allow rule must not create a cross-tenant mTLS bypass.
REVOCATION
CRL and OCSP enforcement required within the platform's stated cache window. Revocation must block new connections and be testable with a cache TTL ≤ 5 minutes for the harness run.
CLOUDFLARE — OPEN ITEM
CA isolation approach must be committed today
Cloudflare's native mTLS model is single-account. Four candidate approaches are in the POC Deliverables v4:

a) Native account-level mTLS isolation (if achievable)
b) Dedicated Cloudflare account per tenant
c) Engineered sub-account CA scoping
d) SI-managed PKI with custom binding

The approach chosen determines how the adversarial test battery (chain manipulation, SAN overlap, shared-root pivot) is instrumented. Harness cannot be finalized without this commitment.
TEST COVERAGE
9 test cases: TC-CR01-001 through TC-CR01-009. Layer 1 positive path + outbound mTLS. Layer 3 adversarial: SAN overlap, chain manipulation, shared-root pivot, cross-tenant cert presentation.
Capability Requirements
CR-02 through CR-06 — Summary
CR-02 · SSH MANAGED ACCESS
18%
Full SSH session termination — not TCP passthrough
Partner authenticates via email + TOTP. Platform injects vault-resident internal credential for backend session. Partner never sees internal credential. SSH traffic must terminate at the proxy — the backend SSH daemon must see platform-internal source IP only.
Cloudflare: SSH proxy and credential vault integration. No structural risk identified — confirm SSH session termination (not passthrough) support.
CR-03 · PER-TENANT DNS REWRITING
10%
Internal FQDNs resolve to SNAT VIP, not real IPs
Each tenant has a structurally dedicated DNS listener VIP. Partner-side DNS query for a CVS internal FQDN must return the tenant's SNAT inbound VIP. Cross-tenant DNS disclosure is tested adversarially.
Cloudflare: Per-tenant CoreDNS forwarding context. Structural isolation of DNS listener per tenant required.
CR-04 · TENANT LIFECYCLE
18%
Provisioning + zero-residual-state offboarding
Tenant provisioning from schema with zero manual steps. Offboarding tested against zero-residual-state: no routing entries, no CA residue, no audit contamination, no reachability to formerly-served endpoints. Run 3× per tenant.
CR-05 · MULTI-TENANT ISOLATION AT SCALE
5%
5 tenants simultaneous, 6 vectors × 20 ordered pairs
Low weight because CR-01 is the structural prerequisite. CR-05 verifies the composition holds with all five tenants active. A single failure in any of the 120 ordered-pair isolation tests fails CR-05 outright.
CR-06 · OUTBOUND GOVERNED ACCESS
10%
CVS-initiated sessions to declared third-party endpoints — same inspection, same audit
Outbound access to declared destinations allowed with TLS inspection active. Undeclared destinations blocked. CVS client certificates presented to the third-party endpoint. Outbound is a first-class governed access mode, not a policy exception or workaround.
CR-07
STRUCTURAL RISK — CLOUDFLARE 8% weight
Third-Party Inline Inspection Plumbing
WHAT WE REQUIRE
A per-tenant steering primitive that routes decrypted in-flight TLS and SSH payload to a CVS-designated per-tenant inspection container (Palo Alto VM-Series vsys-per-tenant in the reference design).

Key-custody stays at the vendor edge. Inspection appliance receives plaintext payload only — no session keys. Per-tenant decrypt-exclusion list supported. Fail-closed is the POC default.
SCOPE — PLUMBING ONLY
CR-07 does not evaluate DLP detection accuracy, signature coverage, or inspection appliance performance. It evaluates whether the vendor platform can deliver the steering contract. Inspection product selection is a separate program.
TEST COVERAGE
12 harness tests (I-01 through I-12): per-tenant steering inbound + outbound, cross-tenant scope isolation, decrypt-exclusion list, fail-closed / fail-open semantics, SSH session inspection, telemetry correlation (≥99% join rate), key custody verification, lifecycle offboarding.
CLOUDFLARE — REALIZATION MUST BE NAMED TODAY
No first-class post-decryption steering primitive
Cloudflare does not provide a native post-decryption steering primitive equivalent to an F5 service-policy chain. Three candidate realizations are in the POC Deliverables v4:

Option A: GENEVE encapsulation via Magic Network — routes decrypted traffic to CVS inspection appliance

Option B: Sub-tenant Workers function as the per-tenant inspection primitive

Option C: Customer-operated Magic WAN on-ramp as traffic intercept

The integrator must name one of these realizations today. The harness test instrumentation for I-01 through I-12 cannot be finalized without knowing the steering architecture.
KEY CUSTODY REQUIREMENT (CR-07.3)
Whichever realization is chosen, the inspection container must receive plaintext payload only. TLS session private keys and key material must be absent from the inspection device's configuration, disk, and memory. This is tested adversarially in I-10 (key custody verification).
